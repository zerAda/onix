## Existing Architecture Baseline (invariants to honour)

The following constraints come from `.planning/codebase/ARCHITECTURE.md` and `AGENTS.md`.
They must not be broken by any hardening integration:

| Invariant | Where defined | Consequence for hardening |
|-----------|--------------|--------------------------|
| Fail-closed: identity error → 401, graph down → 502, ACL error → 403 | `access-gateway/app/` | Backup/restore must not leave the stack in a half-restored state that bypasses gateway checks |
| Cache stores ONLY perimeter-deterministic body; ACL re-applied per request | `access-gateway/app/cache.py` | Observability must NOT instrument cache key internals (contain HMAC material) |
| `num_ctx` wired in Modelfile/compose — do not regress | `docker-compose.yml`, `AGENTS.md` | Backup must preserve `ollama_data` volume intact |
| Ollama reached via internal service name `http://ollama:11434` | `AGENTS.md` §7 piège 1 | Observability sidecars must NOT introduce a new `localhost:11434` alias |
| Telemetry OFF (`DISABLE_TELEMETRY=true`) | `docker-compose.yml` api_server/background | No observability component may phone home; Grafana analytics already disabled in `monitoring/docker-compose.monitoring.yml` |
| Zero secrets in repo; `.env` gitignored, generated by `gen-secrets.sh` | `AGENTS.md` §5 rule 3 | Backup encryption key must itself come from `.env` or a mounted secret, never hardcoded |
| Metric labels must use route templates, never actual user/document values | `access-gateway/app/metrics.py`, `ARCHITECTURE.md` anti-pattern | All new Prometheus instrumentation must follow same rule |
| `runAsNonRoot` in containers | `AGENTS.md` §5 rule 3 | Backup container launched with `--rm` via `docker run` must also run as non-root where image permits |

## Component Boundaries per Hardening Workstream

### WS1 — Backup / Restore Orchestration

**What it touches:**
- `scripts/backup.sh` (already exists — cold backup via `docker compose stop`)
- `scripts/restore.sh` (already exists)
- New: encryption at rest of backup archives, consistency ordering, encryption key management
- New: `deploy/local-prod/` — optional systemd timer for scheduled backup

**What it must NOT touch:**
- `docker-compose.yml` service definitions (backup is external to compose)
- Volume names (already stable: `db_volume`, `opensearch-data`, `minio_data`, `file-system`)
- The `ollama_data` volume is explicitly excluded from backup (re-pullable) — do not change this
- `.env` (read-only consumer; backup encryption key sourced from it, not written to it)

**Integration pattern — existing baseline:**
`backup.sh` today does: `docker compose stop` → tar.gz each volume via ephemeral alpine container → `docker compose start`. This is a cold consistent backup. Ordering is implicit (stop all, archive, start all).

**Integration gap — what the milestone must add:**

1. **Consistency ordering**: The current stop is unordered. To protect Postgres WAL integrity, the stop sequence should drain application services first before stopping the data tier:
   ```
   Stop order: nginx → web_server → api_server → background → access-gateway → actions
   Then: relational_db → opensearch → cache → minio
   Start order: reverse (data tier first, then app tier)
   ```
   Implement via `docker compose stop <service>` calls in sequence in `backup.sh`, or by using `docker compose down --remove-orphans` which respects `depends_on` reverse order only if `restart: always` is off. Safer: explicit ordered stops.

2. **Encryption at rest**: Archives contain Postgres data (secrets in plaintext per CONCERNS.md — Onyx FOSS encryption is a no-op). Encrypt with `openssl enc -aes-256-cbc -pbkdf2` using `BACKUP_ENCRYPTION_KEY` sourced from `.env`. Key generated by `gen-secrets.sh`. Never store key alongside archive.

3. **Postgres-specific**: Before stopping, run `pg_dump` (logical dump via `docker exec relational_db pg_dumpall -U postgres`) in addition to volume snapshot. Logical dump survives Postgres major version changes; volume snapshot does not. Store both.

4. **OpenSearch snapshot API (optional)**: OpenSearch supports a snapshot repository API (`PUT /_snapshot/...`). For consistency, the volume-level backup (after stop) is sufficient and simpler than configuring a snapshot repository. Use volume backup.

5. **Restore verification**: After `restore.sh`, run `scripts/verify.sh` to confirm stack integrity. Add to restore.sh's final step.

**Files to create/modify:**
- `scripts/backup.sh` — add ordered stop, encryption, pg_dump
- `scripts/restore.sh` — add decryption, pg_dump restore path, verification call
- `scripts/gen-secrets.sh` — add `BACKUP_ENCRYPTION_KEY` generation
- `deploy/local-prod/onix-backup.timer` + `onix-backup.service` — systemd timer (weekly)

### WS2 — Healthcheck Ordering + Restart + Startup Credential Guard

**What it touches:**
- `docker-compose.prod-local.yml` (already implements healthchecks + `restart: always` + `depends_on: service_healthy`) — this workstream is LARGELY DONE
- `scripts/preflight-local.sh` (already exists, checks prerequisites)
- New: startup guard rejecting default credentials

**What it must NOT touch:**
- `docker-compose.yml` (base file) — guard is in the prod-local overlay or in preflight
- `access-gateway/app/` — no Python code changes for a startup guard (infra concern)

**Current state (from `docker-compose.prod-local.yml` analysis):**
- `restart: always` — ALL services ✓
- Healthchecks — ALL services ✓ (relational_db via pg_isready, opensearch via curl -k, cache via redis-cli ping, minio already in base, inference_model_server via urllib, ollama already in base, api_server via curl /health, web_server via wget, nginx already in base, actions already in base)
- `depends_on: condition: service_healthy` — api_server, background, web_server, nginx ✓

**Integration gap — credential guard:**

The startup guard must reject production deployment when default credentials are detected. It belongs in `scripts/preflight-local.sh` (already run before `make up-local-prod`), NOT inside compose services or application code.

Logic:
```bash
# In preflight-local.sh — ajout garde anti-secrets-par-défaut
if [ "${POSTGRES_PASSWORD:-}" = "password" ]; then
  echo "ERREUR : POSTGRES_PASSWORD=password détecté — secrets par défaut refusés en production."
  exit 1
fi
if [ "${MINIO_ROOT_PASSWORD:-}" = "minioadmin" ]; then
  echo "ERREUR : MINIO_ROOT_PASSWORD=minioadmin détecté — secrets par défaut refusés en production."
  exit 1
fi
if [ "${OPENSEARCH_ADMIN_PASSWORD:-}" = "admin" ]; then
  echo "ERREUR : OPENSEARCH_ADMIN_PASSWORD=admin détecté — refusé en production."
  exit 1
fi
```

The guard reads `.env` (sourced at shell level in the script) before the compose stack starts. If it fires, `make up-local-prod` aborts before any container starts.

**The `Makefile` target `up-local-prod` must chain:**
```makefile
up-local-prod: preflight-local
    docker compose -f docker-compose.yml -f docker-compose.prod-local.yml up -d
```

**Files to modify:**
- `scripts/preflight-local.sh` — add credential guard section
- `Makefile` — ensure `up-local-prod` depends on `preflight-local`

**Integration gap — only one item missing:**
The `actions` microservice does not yet expose a `/metrics` endpoint. The prometheus job `onix-actions` in `prometheus.yml` already expects it at `actions:8100/metrics`. The gap is in `actions/app/main.py`: add a Prometheus `/metrics` endpoint following the same pattern as `access-gateway/app/metrics.py`.

Metrics to expose (specified in `monitoring/prometheus/prometheus.yml` comments):
- `onix_http_requests_total{endpoint,method,status}` — label `endpoint` = route template (e.g., `/audit`, `/generate`), not per-request path
- `onix_http_request_duration_seconds_bucket{endpoint,le}`
- `onix_killswitch_blocked_total{feature,reason}`
- `onix_budget_spent_eur`, `onix_budget_limit_eur`, `onix_budget_ratio`
- `onix_up`

**PII/cardinality protection (MANDATORY — existing anti-pattern):**
```python
# BON : label = template de route
Counter("onix_http_requests_total", ..., labelnames=["endpoint", "method", "status"])
# L'enregistrement utilise le pattern, pas la valeur réelle :
counter.labels(endpoint="/audit", method="POST", status="200").inc()

# MAUVAIS : ne jamais faire
counter.labels(endpoint=f"/audit/{job_id}", ...)  # cardinalité infinie + PII
counter.labels(user_id=principal.user_id, ...)    # PII directe
```

**Compose integration pattern — separate project, external network:**
The monitoring stack joins `onix-net` as an external network (already implemented in `docker-compose.monitoring.yml`). This is the correct pattern for a single-machine deployment: observability services are peers on the same Docker network, not sidecars. Sidecars would require modifying every service definition and adding complexity; the external-network join achieves the same scraping without touching the application compose file.

**Files to modify:**
- `actions/app/main.py` — add `/metrics` endpoint (prometheus-client, same pattern as gateway)
- No changes to `monitoring/` (already complete)

## Architectural Patterns

### Pattern 1: Compose Overlay (existing, extend)

**What:** `docker-compose.prod-local.yml` adds healthchecks, `restart: always`, and ordered `depends_on` on top of the base compose without modifying it.
**When to use:** Any production hardening that changes service runtime behavior but not service definitions.
**Trade-offs:** Compose overlay `depends_on` replaces (not merges) the base's `depends_on` lists — all upstream dependencies must be re-declared explicitly. Already handled in `docker-compose.prod-local.yml`.

### Pattern 2: External Network Join for Observability (existing, complete)

**What:** The monitoring stack (`docker-compose.monitoring.yml`) declares `onix-net` as `external: true` and joins it. Prometheus and exporters can reach application services by internal DNS name.
**When to use:** Observability as a separate compose project on the same host. Avoids sidecar pattern complexity and never touches application service definitions.
**Trade-offs:** Monitoring stack must start after application stack (onix-net must exist). The systemd unit order enforces this (`After=onix.service`). If monitoring stack starts first, Docker Compose will fail with "network not found" — handled by the unit ordering.

### Pattern 3: Ephemeral Alpine Container for Volume Operations (existing, extend)

**What:** `docker run --rm -v <volume>:/v alpine tar czf ...` — ephemeral container reads a named volume and produces an archive. No persistent container, no compose service added.
**When to use:** Backup/restore of Docker named volumes without modifying compose services. Cold backup (stack stopped) guarantees consistency.
**Trade-offs:** Requires stack downtime during backup. For production with data-tier SLAs, online backup (pg_dump for Postgres, MinIO mc mirror, OpenSearch snapshot API) would allow hot backups, but adds complexity. For a single-machine deployment with tens of users and scheduled maintenance windows, cold backup is sufficient and far simpler to verify.

### Pattern 4: Startup Guard in Preflight Script (new)

**What:** `preflight-local.sh` sources `.env` and checks variable values before `docker compose up`. Fails fast with a human-readable error if default credentials are detected.
**When to use:** Production deployments where accidental use of development defaults must be rejected.
**Trade-offs:** The guard only fires when `make up-local-prod` or `preflight-local.sh` is explicitly called. It does not protect against manually running `docker compose up` without the preflight. Mitigated by RUNBOOK documentation and Makefile convention.

## Anti-Patterns (hardening-specific)

### Anti-Pattern 1: Sidecar Observability Containers

**What people do:** Add Prometheus exporters as `sidecar` containers in `docker-compose.yml` next to each application service.
**Why it's wrong:** Modifies the application compose file, increases review surface, makes the base file environment-specific, and couples observability lifecycle to application lifecycle.
**Do this instead:** Use the existing external-network join pattern (`monitoring/docker-compose.monitoring.yml` joining `onix-net`). Already implemented correctly in this codebase.

### Anti-Pattern 2: Per-Request or Per-Document Metric Labels

**What people do:** `counter.labels(path=request.url.path, user=principal.user_id).inc()`
**Why it's wrong:** Cardinality explosion crashes Prometheus; PII leaks into metrics storage (less secured than audit log).
**Do this instead:** Route template labels only: `counter.labels(endpoint="/audit", method="POST", status="200").inc()`. Documented in `ARCHITECTURE.md` existing anti-pattern "Exposing Metrics with PII". Verified correct in `access-gateway/app/metrics.py`.

### Anti-Pattern 3: Storing Backup Encryption Key in the Backup Archive

**What people do:** Include the encryption key or `.env` file in the backup archive "for convenience."
**Why it's wrong:** Negates the purpose of encryption — anyone with the archive also has the key.
**Do this instead:** Key lives only in `.env` (gitignored, on-disk). Archive is encrypted with this key. Restore requires the original `.env` on the target machine. Document this dependency in `docs/RUNBOOK.md`.

### Anti-Pattern 4: Cold Backup Without Ordered Stop

**What people do:** `docker compose stop` (all services simultaneously) then archive volumes.
**Why it's wrong:** Postgres may not flush WAL cleanly if stopped mid-transaction. OpenSearch may have in-flight shard operations.
**Do this instead:** Stop application services first (nginx → api_server → background → access-gateway → actions), then data services (relational_db → opensearch → cache → minio). The updated `backup.sh` must use explicit sequential `docker compose stop <service>` calls.

### Anti-Pattern 5: systemd `Type=simple` for `docker compose up -d`

**What people do:** Use `Type=simple` for the compose start command.
**Why it's wrong:** `docker compose up -d` returns immediately (daemonizes compose). `Type=simple` would mark the unit active before containers are actually healthy.
**Do this instead:** `Type=oneshot` + `RemainAfterExit=yes` — already correctly implemented in `deploy/local-prod/onix.service`. The unit is "active (exited)" once the `up -d` command completes; `systemctl stop` triggers `ExecStop` cleanly.

## Scaling Considerations (single-machine only — K8s out of scope)

| Concern | Current (tens of users) | If load grows |
|---------|------------------------|---------------|
| Postgres connections | max_connections=150, sufficient for compose setup | Add PgBouncer as compose service if connections saturate |

| Backup duration | Cold backup (stack down ~5min per GB) | Add pg_dump hot path for Postgres to reduce downtime |
